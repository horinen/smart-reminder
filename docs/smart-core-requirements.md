# 智能核心需求文档 - 定时任务2

定时任务2是系统中唯一需要AI智能判断的环节。本文档描述其功能需求，不涉及具体实现方案。

---

## 1. 功能定位

### 1.1 核心职责

定时任务2负责"生产"提醒内容：
- 生成一次性提醒实例
- 调整可重复提醒规则

### 1.2 与定时任务1的分工

| 维度 | 定时任务1 | 定时任务2 |
|------|----------|----------|
| 判断性质 | 确定性 | 智能性 |
| 处理内容 | 消费提醒 | 生产提醒 |
| 是否需要AI | 否 | 是 |

---

## 2. 输入需求

定时任务2需要获取以下信息才能做出智能判断：

### 2.1 项目状态

**需要知道**：
- 有哪些项目
- 每个项目的当前状态（活跃/暂停/完成）
- 每个项目最后更新时间
- 项目的重要程度/优先级（如有）

**用途**：判断哪些项目需要关注

### 2.2 用户日程

**需要知道**：
- 用户近期的日程安排
- 日程类型（重要事项/常规/空闲时间）
- 用户的时间偏好模式（从历史日程中学习到的）

**用途**：判断什么时候提醒更合适

### 2.3 提醒效果

**需要知道**：
- 历史提醒的发送情况
- 用户对提醒的反馈（正面/负面/忽略）
- 响应时间分布
- 提醒效果与时间的关系

**用途**：学习优化提醒策略

---

## 3. 输出需求

### 3.1 生成一次性提醒

**输出内容**：
- 具体的提醒时间点
- 提醒内容
- 关联的项目（如有）

**约束**：
- 格式必须能被定时任务1确定性解析
- 时间点必须明确（不能是模糊的时间范围）

### 3.2 调整可重复提醒

**输出内容**：
- 要调整的提醒规则ID
- 要调整的参数
- 新的参数值

**约束**：
- 调整幅度应有边界（不能剧烈变化）
- 格式必须能被定时任务1确定性解析

---

## 4. 数据访问需求

### 4.1 设计约束

Agent 不能直接读写文件，所有数据操作必须通过脚本执行。

### 4.2 读取需求

定时任务2需要脚本提供以下读取能力：

| 数据 | 用途 |
|------|------|
| 项目状态 | 判断哪些项目需要关注 |
| 用户日程 | 判断什么时候提醒合适 |
| 提醒历史 | 学习优化提醒策略 |

**待定**：脚本输出格式需求（见第5节）

### 4.3 写入需求

定时任务2需要脚本提供以下写入能力：

| 操作 | 说明 |
|------|------|
| 新增一次性提醒 | 写入 pending-reminders.json |
| 调整可重复提醒 | 更新 recurring-reminders.json |

**待定**：脚本输入参数需求（见第5节）

## 5. 脚本接口需求

### 5.1 读取脚本

**输出格式**：纯文本摘要 + Markdown轻量结构

**设计原则**：
- 脚本预处理计算字段（如"距上次更新天数"），AI直接使用
- 脚本不需要时间范围参数（历史数据定期清理）

### 5.2 写入脚本

**一次性提醒参数**：
- `--time`：提醒时间（格式：YYYY-MM-DD HH:mm）
- `--content`：提醒内容

**可重复提醒参数**：
- `--id`：提醒ID
- `--time`：提醒时间
- `--content`：提醒内容

**删除可重复提醒参数**：
- `--id`：要删除的提醒ID

---

## 6. 规则格式需求

定时任务1需要确定性判断，因此定时任务2生成的规则必须是**机器可解析的**。

### 6.1 一次性提醒格式

**格式**：
```json
{"id": "xxx", "time": "2026-03-20 20:00", "content": "项目A已经2天没更新了，要继续吗？"}
```

**字段说明**：
- `id`：唯一标识
- `time`：提醒时间（精确到分钟）
- `content`：提醒内容（具体消息，任务1直接发送）

**特点**：
- 不关联项目ID，内容已包含必要信息
- 发送后删除

### 6.2 可重复提醒格式

**格式**：与一次性提醒一致

**区别**：
- 存放在不同文件（`recurring-reminders.json` vs `pending-reminders.json`）
- 生命周期不同：持久存在，由AI手动删除
