# 智能日程管理助手 - 系统架构设计 v2

## 核心目标

防止个人项目半途而废。AI 根据用户状态智能判断时机，主动提醒用户关注项目进度。

## 设计理念

### 职责分离

**定时任务1（发送）只做确定性判断**：
- 规则中包含所有判断条件
- 判断逻辑是确定性的数学/逻辑运算
- 不需要AI理解语义或进行推理

**定时任务2（学习）承担智能分析**：
- 分析项目状态、日程、提醒效果
- 生成/调整提醒规则
- 这是系统中唯一需要AI智能判断的环节

### 边界约束

**系统不处理用户主动请求提醒**：
- 用户说"提醒我..."不属于本系统职责
- 所有提醒都由系统主动生成
- 用户对话只用于更新业务数据和反馈效果

**用户反馈由 Agent 主动澄清**：
- 用户回复提醒时，Agent 需要判断反馈对应哪个项目/提醒
- 如果不明确，Agent 主动追问澄清

### 状态管理

**无需额外存储已发送状态**：
- 时间窗口定义足够明确，保证不会重复触发
- 例如：一次性提醒发送后即删除；可重复提醒的时间窗口逻辑本身防止同一天重复发送

## 数据文档

### 文档清单

| 文档 | 职责 | 更新者 |
|------|------|--------|
| `projects.json` | 项目信息（状态、进度、备注） | 用户对话 |
| `schedules.json` | 用户日程（保留原始语义） | 用户对话 |
| `pending-reminders.json` | 一次性提醒（预计算实例） | 定时任务2 |
| `recurring-reminders.json` | 可重复提醒（配置规则） | 定时任务2 |
| `reminder-history.json` | 提醒历史 + 效果反馈 | 定时任务1 / 用户对话 |

### 提醒文档设计

#### pending-reminders.json（一次性提醒）

**特点**：预计算的具体提醒实例，发送后删除（历史已记录到 reminder-history.json）

**来源**：定时任务2根据项目状态生成

**规则形式**：给定当前时间，能明确判断"提醒/不提醒"

```
示例：
- 2026-03-20 20:30 提醒用户检查项目X的进度
- 2026-03-21 10:00 提醒用户项目Y即将到期
```

#### recurring-reminders.json（可重复提醒）

**特点**：长期配置规则，每天/每周重复触发

**内容**：
- 触发条件（时间窗口 + 业务条件）
- 提醒内容模板
- 重复周期

```
示例：
- 每个工作日 20:00-22:00，如果项目超过3天未更新，提醒用户
- 每周日晚间，提醒用户回顾所有项目进度
```

#### reminder-history.json（提醒历史）

**职责**：
- 记录所有已发送的提醒
- 记录用户反馈（响应时间、评价等）
- 为定时任务2提供学习数据

## 定时任务设计

### 定时任务1：发送提醒

**职责**：
1. 读取 `pending-reminders.json` 和 `recurring-reminders.json`
2. 执行确定性判断逻辑（比对时间、计算条件）
3. 判断是否发送提醒
4. 发送提醒
5. 写入 `reminder-history.json`
6. 删除已发送的一次性提醒

### 定时任务2：统合学习

**职责**：
1. 读取 `projects.json`、`schedules.json`、`reminder-history.json`
2. 分析项目状态 → 生成新的一次性提醒
3. 分析提醒效果 → 调整可重复提醒的参数
4. 写入 `pending-reminders.json` 和 `recurring-reminders.json`

**智能分析**：这是系统中唯一需要AI进行语义理解和推理的环节
